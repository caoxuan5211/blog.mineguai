ArchRAG 项目理解入门

1. 项目一句话说明

ArchRAG 是一个基于知识图谱的 Graph-based RAG 系统。它的核心目标不是只做“向量检索 + 生成”，而是把语料先构造成图，再把图中的实体与**社区（community）**组织成多层级结构，最后在查询时按层级检索并做答案生成。

这套仓库同时包含两部分：

  1. 论文方法本体的实现层：根目录 src/*.py
  2. GraphRAG / Faiss 的基础设施层：
    • src/graphrag/：Microsoft GraphRAG 的基线与图构建代码
    • HCHNSW/faiss/：基于 Faiss 1.8.0 改造的自定义 ANN 索引实现

可以把它理解成：

GraphRAG 负责把文本变成 KG
ArchRAG 负责把 KG 进一步变成“层次社区索引”
HCHNSW 负责让多层检索真正跑得快

3. 论文在讲什么

论文题目：ArchRAG: Attributed Community-based Hierarchical Retrieval-Augmented Generation
论文链接：https://arxiv.org/abs/2502.09891
我核对到的 arXiv 页面显示最新版本为 v3，2025-08-08。

3.1 论文要解决的三个问题

论文对 GraphRAG 的批评可以概括成三点：

  1. 社区质量低

    • GraphRAG 主要依赖图结构做社区划分
    • 忽略了实体/关系文本属性
    • 导致同一社区里语义可能很杂，社区摘要质量也会受影响

  2. 粒度不灵活

    • 现有方法通常只在某一个粒度上检索
    • 抽象问题和细粒度事实问题很难同时兼顾

  3. 在线 token 成本高

    • 特别是 GraphRAG 的 global search，需要遍历大量社区摘要
    • 在真实问答中成本会非常高

3.2 论文的核心主张

ArchRAG 的回答是：

  1. 用属性增强社区替代纯结构社区
  2. 用层次化社区结构同时覆盖抽象与具体问题
  3. 用C-HNSW / HCHNSW 这样的层次索引降低多层检索成本
  4. 用adaptive filtering + map/reduce减少长上下文直接塞给 LLM 的问题

3.3 论文方法的两阶段

阶段 A：离线索引

  1. 从语料构建 KG
  2. 对 KG 做属性增强与层次聚类
  3. 为每个社区生成 LLM 摘要
  4. 为每一层生成 level summary
  5. 把“社区 + 实体”一起写入 HCHNSW 索引

阶段 B：在线检索与生成

  1. 把问题编码成 query embedding
  2. 在 HCHNSW 上做多层搜索
  3. 同时取出实体、社区、关系、可选 chunk
  4. 先 map，再 reduce，得到最终答案

5. 端到端执行流程

5.1 图构建阶段：先跑 GraphRAG

对应资料：

  • README.md
  • src/graphrag/README.md
  • dataset/settings.yaml

流程大致是：

  1. 把原始语料放进 corpus/input
  2. 用 GraphRAG 从文本 chunk 中抽实体和关系
  3. 生成 create_final_entities.parquet 与 create_final_relationships.parquet
  4. 把它们作为 ArchRAG 的输入图

注意：

  • dataset/settings.yaml 里显式跳过了一些 GraphRAG 工作流，例如 create_final_community_reports
  • 说明这个项目只把 GraphRAG 当成 KG 构建器，而不是直接使用它自带的社区摘要/检索流程

5.2 ArchRAG 离线建索引

主入口：src/index.py

代码级步骤

  1. read_graph_nx()

    • 从 entity / relationship 文件读取图
    • 位置：src/utils.py

  2. process_entity_embedding()

    • 给实体算 embedding，并挂到 networkx 图节点上

  3. attr_cluster()

    • 位置：src/attr_cluster.py
    • 做多层聚类

  4. create_hchnsw_index()

    • 位置：src/hchnsw_index.py
    • 把多层社区和实体一起写入 HCHNSW

  5. make_level_summary()

    • 位置：src/index.py
    • 实际调用 src/client_reasoning.py -> level_summary()

  6. 保存多个中间/最终文件

    • community_df_intermediate.csv
    • community_df_index.csv
    • entity_df_index.csv
    • relationship_df_index.csv
    • relationship_embedding.csv
    • level_summary.csv
    • hchnsw.index

核心理解

ArchRAG 的索引对象不是单一向量库，而是三层东西一起组成的：

  1. 实体
  2. 社区
  3. 层摘要

在线问答时真正起作用的是它们的组合，而不是某一个单独对象。

5.3 在线问答

主入口：src/inference.py

整体函数链：

load_index()
-> hcarag()
-> hcarag_retrieval()
-> hcarag_inference()
-> hcarag_inference_direct() 或 hcarag_inference_mr()

检索阶段做了什么

  1. 对 query 做 embedding
  2. 根据策略决定每层取多少个结果
    • global
    • all
    • inference
  3. 按层调用 HCHNSW 搜索
  4. 合并得到候选实体/社区
  5. 从候选实体关联出的关系中，再按 embedding 相似度取 top-k 关系
  6. 可选：
    • 取 vanilla chunk 检索结果
    • 做 PPR refine

生成阶段做了什么

有两条路线：

  1. direct

    • 直接把实体/关系/社区拼 prompt，让 LLM 出答案

  2. mr

    • 先让 LLM 对问题做一步自由推理
    • 再把检索结果拆成多个 chunk 做 map
    • 每个 chunk 返回 points
    • 最后 reduce 汇总成答案

论文里说的 adaptive filtering，在代码里主要体现在这条 mr 路径上。

6. 核心模块逐个解读

6.1 src/utils.py：底座工具文件

这个文件很重要，承担了四类职责：

  1. 参数解析

    • create_arg_parser()
    • create_inference_arg_parser()

  2. 图读取

    • read_graph_nx()
    • 将实体和关系文件转成 NetworkX 图

  3. embedding

    • entity_embedding()
    • relation_embedding()
    • 通过 OpenAI-compatible API 或本地模型算向量

  4. 图增强

    • compute_distance()
    • 先把原图边赋权为余弦相似度
    • 再用 HNSW 找相似节点补边

这里体现的论文思想

论文里“属性增强”的本质，在这份代码里首先体现为：

  • 把实体文本变成 embedding
  • 再用 embedding 相似度补图边与赋边权

也就是说，这个仓库目前对“attribute”最直接的实现就是文本 embedding 驱动的图增强。

6.2 src/attr_cluster.py：层次聚类主逻辑

这是 ArchRAG 的算法核心。

主函数：attr_cluster()

每一层循环都做：

  1. 若 augment_graph=True，先 compute_distance() 得到增强图
  2. 用聚类算法得到社区
    • 默认 weighted_leiden
    • 也支持 spectral clustering
  3. 若不是第一层，把低层社区重新编号并映射到上层
  4. 为社区生成 LLM 报告
  5. 根据新社区重建图
  6. 进入下一层

关键设计

这个实现不是“先一次性做完层次聚类，再统一摘要”，而是：

一层聚类 -> 一层摘要 -> 用摘要/社区结果重构上层图 -> 再聚类

这正对应论文里“LLM-based iterative hierarchical clustering framework”的叙事。

终止条件

  • 达到 max_level
  • 或者当前层聚类数低于 min_clusters

6.3 src/community_report.py：社区报告生成

职责：

  1. 根据社区里的实体、关系、下级社区拼装文本
  2. 调 LLM 生成结构化 JSON 报告
  3. 解析出：
    • title
    • summary
    • findings
    • rating
    • rating_explanation
  4. 再给报告文本算 embedding

为什么重要

社区不只是一个 cluster id。
在 ArchRAG 里，社区被提升成了可检索的语义对象。

这意味着社区有两种表示：

  1. 图结构表示
  2. 文本摘要表示

后续检索时真正喂给 LLM 的，主要是第二种。

6.4 src/client_reasoning.py：查询理解与生成编排

这份文件承担两类很重要的逻辑：

A. 查询如何决定搜哪一层

problem_reasoning()

它会让 LLM 读取：

  • 每层的 level summary
  • 每层社区数量
  • 每层随机社区例子

然后输出每层的重要性分数，再转换成 k_per_level。

也就是说，论文中的“支持抽象问题与具体问题的分层检索”在代码里不是硬编码规则，而是LLM 决策的层级预算分配。

B. 检索后怎么生成答案

这里实现了：

  • prep_e_r_content()
  • prep_community_content()
  • map_inference()
  • reduce_inference()

它本质上是一个面向 token 限额的证据整理器。

6.5 src/hchnsw_index.py：层次索引写入

职责很清晰：

  1. 读取社区向量和实体向量
  2. 把社区 level 与实体 level 合成 combined_levels
    • 社区是 1..L
    • 实体是 0
  3. 调 faiss.IndexHCHNSWFlat
  4. 调 set_vector_level(levels)
  5. add() 所有向量

这里可以把它理解成：

把“多层社区 + 底层实体”压到一个支持层级搜索的 ANN 索引里。

6.6 src/inference.py：在线检索主链

几个必须记住的函数：

  • hcarag()
  • hcarag_retrieval()
  • load_strategy()
  • hcarag_inference_mr()

load_strategy() 的三种检索策略

  1. global

    • 每层取固定 k_each_level

  2. all

    • 更接近 GraphRAG 风格，扩大低层取样范围

  3. inference

    • 先让 LLM 判断层级重要性
    • 再按比例分配检索预算

这说明论文里的“adaptive hierarchical retrieval”在仓库中至少有一个明显落地点：strategy=inference。

7. HCHNSW 到底是什么

对应目录：HCHNSW/faiss/

7.1 它不是普通 HNSW

Faiss 原生提供的是 IndexHNSW*。
这个项目新增的是 IndexHCHNSW*，核心特征有两个：

  1. 每个向量都有 level
  2. 搜索时可以指定 search_level

相关文件：

  • HCHNSW/faiss/faiss/IndexHCHNSW.h
  • HCHNSW/faiss/faiss/IndexHCHNSW.cpp
  • HCHNSW/faiss/faiss/impl/HCHNSW.h
  • HCHNSW/faiss/faiss/impl/HCHNSW.cpp

7.2 从头文件看它提供了什么

SearchParametersHCHNSW 里新增了：

  • search_level
  • entry_point
  • efSearch

HCHNSW 结构里新增了：

  • levels
  • cross_neighbors
  • first_entry_points_in_level

这说明它不是单纯把所有向量混在一个图里搜索，而是明确维护了层级信息和跨层入口。

7.3 搜索过程怎么理解

从 impl/HCHNSW.cpp 的 HCHNSW::search() 看，大致逻辑是：

  1. 从最高层入口点出发
  2. 对高于目标层的各层做 greedy search
  3. 沿 cross_neighbors 逐层下降
  4. 到达目标层后再做该层近邻搜索

这和论文叙述的“分层查找 + 逐层下钻”是一致的。

7.4 这个模块在项目里的意义

如果没有 HCHNSW，ArchRAG 只能：

  1. 每层单独建一个 ANN 索引
  2. 或者完全不用 ANN，直接暴力遍历

那样：

  • 工程实现更碎
  • 多层检索切换更慢
  • 不能自然支持跨层 entry point 传递

所以 HCHNSW 是论文“效率提升”主张的关键支撑。

9. 论文结论与仓库落地的对应关系

基于论文与代码，我建议这样理解“论文贡献是否真的落地”：

9.1 已明显落地的部分

  1. 属性增强聚类

    • 代码里通过 embedding 图增强 + weighted clustering 实现

  2. 层次社区摘要

    • 社区报告与 level summary 都有实现

  3. 多层索引

    • IndexHCHNSWFlat 已落地

  4. 多层检索

    • 支持 per-level search 和层级预算

  5. adaptive filtering 风格生成

    • map + reduce 已有代码

9.2 更像“论文思路”，但工程上还比较粗糙的部分

  1. 查询理解依赖 LLM 打分，稳定性仍受提示词影响
  2. 很多路径使用固定阈值和启发式参数
  3. 数据路径、API 路径、实验路径大量写死
  4. 离线/在线流程耦合较重，复现实验门槛不低

10. 这个仓库和原版 GraphRAG 的关系

可以这样记：

GraphRAG 提供了什么

  1. 文本切 chunk
  2. 从文本抽实体和关系
  3. 基础图工作流

ArchRAG 在什么地方超出了 GraphRAG

  1. 不满足于纯结构社区
  2. 引入了多层社区迭代构建
  3. 引入了专门的层次 ANN 索引
  4. 不只做 global/local 二分，而是多层联合检索
  5. 生成阶段强调 adaptive filtering，而不是简单把所有检索结果拼接

11. 当前代码的工程现状与问题

这是做项目接手时必须知道的现实情况。

11.1 可移植性一般

很多脚本和评测代码里存在硬编码路径，例如：

  • /mnt/data/wangshu/...
  • 固定数据集目录
  • 固定 index 文件位置

这意味着：

  • 仓库不是开箱即用
  • 复现依赖原作者的目录组织方式

11.2 配置分散

配置来源同时分布在：

  • dataset/*.sh
  • dataset/settings.yaml
  • src/utils.py 的 argparse 默认值

这会导致参数真实生效位置不够直观。

11.3 密钥与实验参数管理不规范

样例脚本中出现了硬编码 API 配置位，说明项目还停留在实验脚本风格，而不是正式产品风格。

11.4 测试不足

虽然有 src/evaluate/，但更偏实验评测，不是单元测试/集成测试体系。

11.5 模块边界还可以更清晰

例如：

  • utils.py 过于臃肿
  • prompt、推理、数据处理有明显交叉
  • 线上推理和实验脚本耦合较深

14. 论文实验结论可怎么理解

结合论文摘要、方法段落与结果页，可以提炼出三点：

  1. 准确性更好

    • 论文声称在 specific QA 上优于已有 graph-based RAG

  2. token 成本更低

    • 关键原因是不用像 GraphRAG global search 那样遍历全部社区

  3. 社区质量更高

    • 论文给出了 attributed clustering 相比 Leiden 的社区质量提升

对照仓库实现，这些结论分别对应：

  1. src/evaluate/test_qa.py
  2. 多层检索 + map/reduce 过滤
  3. metric/ 与 community_evaluate.py

15. 对项目的总体判断

如果从“科研原型”角度看，这个项目的亮点很明确：

  1. 用社区层次替代单层图检索
  2. 用属性增强改善社区质量
  3. 用 HCHNSW 把层次检索结构化落地

如果从“工程产品”角度看，它还有明显短板：

  1. 配置与路径硬编码较多
  2. 依赖外部环境较重
  3. 评测多、测试少
  4. 模块抽象仍偏实验脚本

所以最准确的定位是：

这是一个研究导向、方法亮点强、工程化程度中等的 GraphRAG 改造项目。

16. 一页总结

如果只记住最核心的内容，请记这四句：

  1. ArchRAG 的本质，是把 KG 上的“实体 + 社区”做成层次可检索结构。
  2. 离线阶段的关键是：图增强、层次聚类、社区摘要、HCHNSW 建索引。
  3. 在线阶段的关键是：多层检索、关系补充、adaptive filtering 生成。
  4. 这个仓库最有研究价值的部分不在 src/graphrag/，而在根级 src/*.py 和 HCHNSW。
